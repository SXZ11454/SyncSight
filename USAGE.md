# SyncSight P2P 屏幕共享 - 使用指南

## 功能特性

✅ **已实现**
- 基于 WebRTC 的 P2P 音视频传输
- 未加密 HTTP 信令服务器（局域网使用）
- 多客户端支持（1 个发送端 + 多个接收端）
- 屏幕/窗口/摄像头共享
- 系统音频 + 麦克风音频混合
- 自动 STUN 服务器配置
- 房间管理系统
- MDUI 2 美观界面

## 快速开始

### 1. 启动发送端

```bash
npm start
```

程序将：
- 自动启动 P2P 信令服务器（默认端口 3000）
- 创建房间（8 位房间号）
- 显示访问地址和房间号

### 2. 选择共享源

**三种共享模式**：
- 📺 **屏幕** - 共享整个显示器
- 🪟 **窗口** - 共享特定应用程序窗口
- 📷 **摄像头** - 共享摄像头画面

**共享选项**：
- ✅ 共享画面 - 启用视频传输
- ✅ 共享系统声音 - 捕获系统音频
- ✅ 共享麦克风 - 捕获麦克风输入
- 🎤 选择麦克风 - 从下拉框选择特定麦克风

### 3. 开始共享

1. 选择要共享的源（屏幕/窗口/摄像头）
2. 配置共享选项
3. 点击"开始共享"按钮
4. 记录显示的访问地址和房间号

示例输出：
```
http://192.168.3.100:3000 (房间：A1B2C3D4)
```

### 4. 接收端加入

**直接访问房间**：
1. 打开浏览器（Chrome/Edge/Firefox）
2. 访问：`http://<发送端 IP>:3000/room/<房间号>`
   - 例如：`http://192.168.3.100:3000/room/A1B2C3D4`
3. **视频将自动播放**（默认静音）

**通过加入页面**：
1. 打开浏览器
2. 访问：`http://<发送端 IP>:3000/join`
3. 输入房间号（任意长度）
4. 输入昵称（可选）
5. 点击"加入房间"
6. **视频将自动播放**（默认静音）

### 5. 播放视频流

接收端页面将：
- 自动连接到信令服务器
- 建立 P2P 连接
- 接收视频流
- **自动播放视频**（默认静音）

**注意**：
- ✅ 视频会自动播放
- 🔇 默认静音（浏览器自动播放策略）
- 📱 移动端可能需要用户交互才能播放

## 多设备连接

### 连接流程

```
发送端 (Host)          信令服务器          接收端 1 (Client)
    |                       |                      |
    |-- 创建房间 ---------->|                      |
    |                       |<-- 加入房间 --------|
    |<-- CLIENT_JOINED -----|                      |
    |                       |                      |
    |-- OFFER ------------->|                      |
    |                       |-- OFFER ------------>|
    |                       |                      |
    |                       |<-- ANSWER -----------|
    |<-- ANSWER ------------|                      |
    |                       |                      |
    |    === P2P 连接建立，直接传输媒体流 ===        |
    |<============================================>|
```

### 支持的设备数量

- **理论上限**：无限制
- **推荐上限**：10 个接收端
- **带宽考虑**：每个客户端独立 P2P 连接，发送端上行带宽 = 码率 × 客户端数量

### 客户端管理

发送端日志将显示：
- ✅ 客户端加入通知
- ✅ 客户端断开通知
- ✅ 连接状态变化
- ✅ 视频质量调整（未来）

## 网络配置

### 局域网访问

发送端将自动检测并显示局域网 IP 地址：
- ✅ 优先使用非链路地址（非 169.254.x.x）
- ✅ 多网卡自动选择第一个可用 IP
- ⚠️ 确保防火墙开放端口（默认 3000）

### 跨网络访问

当前配置使用公共 STUN 服务器：
```javascript
iceServers: [
  { urls: 'stun:stun.l.google.com:19302' },
  { urls: 'stun:stun1.l.google.com:19302' },
  { urls: 'stun:stun2.l.google.com:19302' },
  { urls: 'stun:stun3.l.google.com:19302' },
  { urls: 'stun:stun4.l.google.com:19302' }
]
```

**穿透能力**：
- ✅ 同一局域网内：100% 成功
- ✅ 普通 NAT：90% 成功
- ⚠️ 对称 NAT/企业防火墙：可能需要 TURN 服务器

### 配置 TURN 服务器（可选）

如需支持复杂网络，编辑 `src/renderer/renderer.js`：

```javascript
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:your-turn-server.com:3478',
      username: 'your-username',
      credential: 'your-password'
    }
  ]
});
```

推荐 TURN 服务：
- [coturn](https://github.com/coturn/coturn) - 开源自建
- [Twilio](https://www.twilio.com/stun-turn) - 付费云服务

## 故障排查

### 1. 发送端无法启动

**错误**：`EADDRINUSE: address already in use :::3000`

**解决**：
1. 更换端口（在发送端界面修改端口号）
2. 或使用命令查找占用进程：
   ```bash
   netstat -ano | findstr :3000
   taskkill /PID <PID> /F
   ```

### 2. 接收端无法连接

**错误**：`无法连接到服务器`

**检查**：
1. ✅ 确认发送端已启动并显示访问地址
2. ✅ 确认 IP 地址正确（非 127.0.0.1）
3. ✅ 确认端口号正确
4. ✅ 检查防火墙设置
5. ✅ 确认在同一局域网或网络可达

**测试**：
```bash
# 从接收端设备 ping 发送端
ping <发送端 IP>

# 测试端口连通性
telnet <发送端 IP> 3000
```

### 3. P2P 连接失败

**错误**：`连接失败` 或 `ICE 连接失败`

**原因**：
- NAT 类型不兼容
- 防火墙阻止 P2P 连接
- STUN 服务器不可达

**解决**：
1. 尝试在同一局域网内测试
2. 检查浏览器 WebRTC 设置：
   - Chrome: `chrome://webrtc-internals`
   - Firefox: `about:webrtc`
3. 配置 TURN 服务器（见上文）

### 4. 无视频流

**错误**：`No screen stream available`

**检查**：
1. ✅ 确认已选择共享源
2. ✅ 确认已授予屏幕录制权限
3. ✅ 检查源是否被其他程序占用

**权限设置**：
- **macOS**: 系统偏好设置 > 安全性与隐私 > 隐私 > 屏幕录制
- **Windows**: 设置 > 隐私 > 应用权限 > 屏幕录制
- **Linux**: 取决于桌面环境

### 5. 无音频

**检查**：
1. ✅ 确认已勾选"共享系统声音"或"共享麦克风"
2. ✅ 确认浏览器未静音
3. ✅ 检查系统音频设置

**麦克风问题**：
1. 确认已选择正确的麦克风
2. 授予麦克风权限
3. 检查麦克风是否被其他程序占用

### 6. 视频卡顿

**优化建议**：
1. 降低帧率（编辑代码中的 `maxFrameRate`）
2. 降低分辨率
3. 减少客户端数量
4. 使用有线网络连接
5. 关闭其他占用带宽的程序

## 安全注意事项

⚠️ **重要警告**

当前实现使用**未加密的 HTTP 传输**：

### 风险
1. **信令明文** - 房间号、连接信息可被截获
2. **无身份验证** - 任何人可加入房间（如果知道房间号）
3. **中间人攻击** - 局域网内可能被劫持
4. **WebRTC 加密** - 媒体流本身有 DTLS 保护

### 建议
- ✅ 仅在受信任的局域网使用
- ✅ 使用复杂的房间号
- ✅ 定期更换房间
- ⛔ 不要在公共网络使用
- ⛔ 不要传输敏感信息

### 未来改进
- [ ] HTTPS + WSS 加密传输
- [ ] 房间密码验证
- [ ] 端到端加密
- [ ] 用户身份认证

## 常见问题

**Q: 支持多少个客户端？**
A: 理论无限制，推荐≤10 个。每个客户端独立 P2P 连接，发送端带宽 = 码率 × 客户端数。

**Q: 可以跨互联网使用吗？**
A: 可以，但需要：
1. 发送端有公网 IP 或正确配置端口转发
2. 或使用 TURN 服务器中继

**Q: 为什么不用 HLS/DASH？**
A: WebRTC 提供超低延迟（<500ms），适合实时互动场景。

**Q: 支持移动端吗？**
A: 接收端支持移动浏览器。发送端需要 Electron（桌面端）。

**Q: 可以录制吗？**
A: 接收端可使用浏览器扩展或屏幕录制软件录制。

**Q: 支持音频吗？**
A: 支持系统音频 + 麦克风音频混合传输。
